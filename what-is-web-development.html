import turtle, math

t = turtle.Turtle()
t.speed(0)

cx, cy = 0, 0
spokes = 8
rings = 5
spoke_length = 150

# draw spokes
for i in range(spokes):
    angle = math.radians(360 / spokes * i)
    t.penup()
    t.goto(cx, cy)
    t.pendown()
    t.goto(cx + spoke_length * math.cos(angle),
           cy + spoke_length * math.sin(angle))

# draw rings
for r in range(1, rings + 1):
    dist = spoke_length / rings * r
    points = []
    for i in range(spokes):
        angle = math.radians(360 / spokes * i)
        points.append((cx + dist * math.cos(angle),
                        cy + dist * math.sin(angle)))
    t.penup()
    t.goto(points[0])
    t.pendown()
    for p in points[1:]:
        t.goto(p)
    t.goto(points[0])

turtle.done()

// a history of productive tensions
STAGE_01 //

GOD IS DEAD. SO IS JQUERY.

You need to understand why both statements mean the same thing.

Technologies do not die because something better arrives. They die because the problem they were solving changed shape. jQuery did not fail. It solved exactly what it was built for — until the problem evolved past it. Every framework you use today will follow the same path. The only question is whether you understand the problem well enough to see it coming.

Most developers do not. Not because they are careless — but because nobody showed them where to look. Tutorials start at chapter three. This starts at chapter one.

CERN. Geneva. 1989. The most brilliant physicists on Earth — working in the same building, breathing the same air. Each producing extraordinary work. Each storing it inside a machine no one else could read. Not classified. Not protected. Just invisible. When a researcher left the institution, their knowledge left with them.

Tim Berners-Lee was not trying to invent the Web. He was trying to solve something quieter — knowledge that existed but could not be found. His question was simple: what if every document had an address, a format any machine could read, and a way to link to any other document on Earth?

How do we give a document an address — and how does a machine know how to ask for it?
STAGE_02 //

THE UNIVERSAL LANGUAGE

The answer was HTML. Not a programming language — a markup language. A system of meaning applied to text. An <h1> does not say "make this big." It says "this is the most important heading in this document." The appearance is secondary. The meaning is primary. This distinction will matter later. File it away.

<html>
  <head>
    <title>Information Management: A Proposal</title>
  </head>
  <body>
    <h1>A system for sharing information across machines.</h1>
    <p>Any machine. Any operating system. Same result.</p>
    <a href="next.html">Continue reading</a>
  </body>
</html>

The isolation was broken. Documents were now reachable — in principle. But a document in Geneva still needed a way to travel to a machine in Tokyo. HTML solved the format. It did not solve the delivery.

We have a universal document format. Now — how does a machine request one across a network?
STAGE_03 //

THE RECEPTIONIST

Imagine a library. Every document humanity has ever written, stored inside. You are outside. You cannot enter. There is a receptionist at the door. You describe what you need. The receptionist searches. The receptionist returns with the document. HTTP is the language you and the receptionist speak.

You never touch the shelves. You send a request. You receive a response. The receptionist does not remember your previous visit unless you remind them. That last sentence is not a detail — it is architecture.

GET /proposal.html HTTP/1.1
Host: info.cern.ch

--- server responds ---

HTTP/1.1 200 OK
Content-Type: text/html

<html>...</html>

Text. Over a wire. That is all it ever was. Every framework you have used — Django, Rails, Express — every single one ends here. Do not believe it?

// strip the abstraction — this is your entire framework
#include <stdio.h>
#include <string.h>
#include <unistd.h>
#include <arpa/inet.h>

int main() {
    int fd = socket(AF_INET, SOCK_STREAM, 0);
    struct sockaddr_in addr = {
        .sin_family = AF_INET,
        .sin_addr.s_addr = INADDR_ANY,
        .sin_port = htons(8080)
    };
    bind(fd, (struct sockaddr*)&addr, sizeof(addr));
    listen(fd, 3);
    int client = accept(fd, NULL, NULL);

    char response[] =
        "HTTP/1.1 200 OK\r\n"
        "Content-Type: text/html\r\n\r\n"
        "<h1>Hello, Web.</h1>";

    write(client, response, strlen(response));
    close(client);
    return 0;
}

Compile it. Open localhost:8080. Your browser — with its millions of lines of code — will render an <h1> served by 30 lines of C. The protocol has not changed since 1991. The complexity you work with every day is built entirely on top of this moment.

Requests and responses work. But every response is a static file. What if machines need to exchange structured data — not just human-readable pages?
STAGE_04 //

SHOULDERS OF GIANTS

Berners-Lee did not invent structure from nothing. He inherited it. Since the 1960s, publishers, defence contractors, and librarians had been solving the same problem — how do you describe a document's structure so that any system can process it? Their answer was SGML. The Web's answer was XML.

<!-- HTML describes presentation -->
<p>Temperature: 36.6</p>

<!-- XML describes meaning -->
<measurement>
  <type>temperature</type>
  <value unit="celsius">36.6</value>
</measurement>

The Web is not a creation story. It is an inheritance story. RSS, SOAP, early web services, and eventually JSON all descend from this idea. The developer who knows this reads any new data format and immediately asks the only question that matters: what problem does this solve that the previous format could not?

Machines can exchange structured data. But the page is still inert. The user can read — but cannot act. What if interaction is required?
STAGE_05 //

THE PAGE WAKES UP

In 1995, Brendan Eich was given ten days to build a programming language for the browser. The result was JavaScript — imperfect, rushed, and completely transformative. For the first time, a document could respond to the person reading it. The browser was no longer a window. It was a runtime.

// before: reload the entire page to change anything
// after:

const button = document.querySelector('#run');
const output = document.querySelector('#output');

button.addEventListener('click', () => {
    output.textContent =
        `Process started at ${new Date().toISOString()}`;
});

No reload. The page mutated in place. This was not a small improvement — it was a categorical shift in what the Web could be. But every action requiring server data still forced a full page reload. The wire had to carry the entire document again. Every time.

Pages are interactive. But server communication still reloads everything. The experience is broken. There must be a way to contact the server silently, in the background.
STAGE_06 //

THE SILENT REQUEST

In 2004, Gmail launched. You received an email without the page reloading. This felt like magic. It was AJAX — the browser sending requests to the server in the background, updating only the part of the page that needed to change. The document-sharing system was now an application platform.

// the page does not reload. the server is contacted silently.

async function loadMessages() {
    const res  = await fetch('/api/messages');
    const data = await res.json();

    document.querySelector('#inbox').innerHTML =
        data.messages
            .map(m => `<div class="msg">${m.subject}</div>`)
            .join('');
}

The receptionist was now working in the background. The experience of the Web changed permanently. And then — quietly — the applications became enormous. State was everywhere. DOM manipulation scattered across thousands of lines. Nobody had planned for this codebase. Nobody could read it.

The browser is now a full application runtime. The complexity is unmanageable. How do we impose order on something that was never designed to be this large?
STAGE_07 //

THE CURRENT THESIS

Frameworks arrived to manage the complexity — React, Vue, Angular. APIs separated concerns. SPAs replaced page navigation with state transitions. Cloud infrastructure decoupled servers from physical machines. The Web became a distributed operating system built on top of a document-sharing protocol from 1991.

Every layer of abstraction you work with daily is a mitigation. Not a solution. React did not solve the complexity problem — it renegotiated it. Next.js renegotiated React's limitations. The tension does not disappear. It changes shape. This is not pessimism. This is the operating condition of every technology that has ever existed.

The developer who understands this does not panic when a framework becomes obsolete. They ask the only question that matters: what problem was this solving, and has that problem changed shape?

Abstraction managed the complexity — but abstraction hides the tension that produced it. Is Next.js a new framework — or is it a socially acceptable return to PHP? The answer depends entirely on whether you understand what problem both were trying to solve.
STAGE_08 //

BACK TO CERN

Berners-Lee's original proposal carried a note from his manager in the margin: "Vague but exciting." He approved it anyway.

The document Berners-Lee was trying to make findable in 1989 is now reachable from any device on Earth in under 200 milliseconds. The isolation was renegotiated — seven times, in the sections you just read — until the original problem became unimaginable.

You are now the next negotiation. The frameworks you use are the current thesis. Their antithesis is already forming — in the abstraction layers that prevent developers from reasoning beneath them, in the specifications followed without understanding the tension that produced them, in the tools mastered without knowledge of the problems they were built to solve.

The most dangerous developer in any codebase is not the one who does not know the framework. It is the one who only knows the framework.

You now know where the framework ends. Everything beneath it — the socket, the wire, the text, the protocol — has been waiting for you to look.

You now know where to look.
